\begin{abstract}
\noindent
Heterogeneous GPU clusters present a precision mismatch problem that limits distributed LLM training and inference. This update presents the RDMA Tensor Cache, a centralized precision management layer for a two-tower home lab: Tower~1 (Ryzen~9~7950X, RTX~5070~Ti~16GB, Windows) and Tower~2 (Proxmox/Linux, $2\times$ Tesla~V100~32GB). We resolve the V100 (FP16-only) vs.\ 5070~Ti (BF16-native, no kernel RDMA) asymmetry with an RDMA-accessible FP32 master weight store, stochastic rounding, predictive tensor prefetching, and a centralized KV cache eliminating per-rank duplication under MLA/MQA tensor parallelism. We introduce GPU pipeline optimization via Amdahl's and Gustafson's scaling laws, analyze CUDA vs.\ Triton kernel abstraction hierarchies, and describe vLLM semantic routing for heterogeneous clusters. Two new tools---ScuffedKernels (RDMA-aware kernel optimization) and ScuffedSearch (autonomous RDMA research automation)---provide systematic frameworks for kernel-level performance engineering and experiment tracking. Integration paths through UCX/UCC, SoftRoCEv2, and UEC~1.0 are described.
\end{abstract}

\section{Overview}
\label{sec:overview}

This update advances the thesis in three directions. First, we formalize the RDMA Tensor Cache architecture that bridges the precision gap between V100 and 5070~Ti GPUs by centralizing master weights in the tensor database's \texttt{tensor\_rdma} crate. For a tensor of size $S$ bytes, the total transfer time over RDMA is:
\begin{equation}
    T_{\text{RDMA}} = T_{\text{latency}} + \frac{S}{B_{\text{link}}} + T_{\text{DMA}}
\end{equation}
where $T_{\text{latency}} \approx 2~\mu$s (hardware RoCE), $B_{\text{link}} = 12.5$~GB/s (100GbE), and $T_{\text{DMA}} \approx 0.5~\mu$s. For a 256~MB gradient tensor, $T_{\text{RDMA}} \approx 20.5$~ms, dominated by bandwidth.

\subsection{The MLA/MQA KV Cache Problem}
\label{sec:mla-problem}

Multi-Latent Attention (MLA, DeepSeek-V2~\cite{deepseekv2}) and Multi-Query Attention (MQA) share a single key-value head across all attention heads, creating a structural incompatibility with tensor parallelism: the KV cache cannot be sharded along the head dimension. Under TP, each rank stores a \emph{full copy}:
\begin{equation}
    \text{KV Memory (per rank, MQA)} = 2 \cdot L \cdot C \cdot d_{\text{kv}} \quad \text{(no reduction with TP)}
\end{equation}

\begin{challenge}
MLA and MQA models cannot benefit from tensor parallelism for KV cache distribution. Each TP rank stores a full copy, negating the memory savings these attention variants provide.
\end{challenge}

Our solution externalizes the KV cache to the tensor database, accessed via RDMA. A single centralized copy serves all TP ranks through one-sided RDMA reads, reducing total KV memory from $\text{TP} \times 2LC \cdot d_{\text{kv}}$ to $2LC \cdot d_{\text{kv}} + \text{TP} \times \text{prefetch\_buffer}$.

\subsection{Precision Pipeline}

FP16 (V100 Tensor Cores) has 5~exponent bits and range $[6 \times 10^{-8},\; 65504]$; gradients routinely exceed this range. BF16 shares FP32's 8~exponent bits but is unavailable on V100. The RDMA Tensor Cache resolves this through a three-stage precision pipeline (\Cref{fig:precision-pipeline}): V100 FP16 gradients $\to$ FP32 master weights in tensor database $\to$ stochastic rounding to device-native format (BF16 for 5070~Ti, FP16 for V100).

\subsection{Stochastic Rounding}
\label{sec:stochastic-rounding}

Stochastic rounding eliminates the systematic bias of round-to-nearest-even. For an FP32 value $x$ rounded to a lower precision with adjacent representable values $\lfloor x \rfloor$ and $\lceil x \rceil$:
\begin{equation}
\label{eq:stochastic-round}
    P(\text{round up}) = \frac{x - \lfloor x \rfloor_{\text{fp16}}}{\lceil x \rceil_{\text{fp16}} - \lfloor x \rfloor_{\text{fp16}}}
\end{equation}
giving the unbiasedness property $\mathbb{E}[\text{round}(x)] = x$, ensuring gradient accumulation converges to the correct value despite per-step precision loss~\cite{micikevicius2018mixed}. The implementation in \texttt{rounding.rs} uses the \texttt{half} crate for FP16/BF16 manipulation and uniform random draws to decide rounding direction.

\subsection{Centralized KV Cache}

The MLA/MQA duplication problem (\Cref{sec:mla-problem}) is resolved by storing the KV cache in the tensor database's \texttt{RdmaTensorCache}, keyed by layer index and sequence position. Each TP rank issues one-sided RDMA reads rather than maintaining a local copy. The prefetch engine predicts needed entries based on the current decoding step, and new KV entries are written back via RDMA Write.

\subsection{Device Capability Model}

The \texttt{DeviceCapability} struct encodes GPU capabilities, and \texttt{PrecisionRouter} selects wire formats. \section{GPU Pipeline Optimization and Scaling Laws}
\label{sec:scaling-laws}
%=============================================================================

Optimizing GPU kernels for RDMA-aware workloads requires principled prioritization. We apply classical scaling laws to determine which kernels yield the greatest whole-program speedup.

\subsection{Amdahl's Law Applied to Kernel Optimization}

Amdahl's Law~\cite{amdahl1967} gives the maximum speedup when accelerating a fraction $p$ of execution by factor $n$:
\begin{equation}
\label{eq:amdahl}
    S(n) = \frac{1}{(1-p) + \frac{p}{n}}
\end{equation}

This directly informs kernel prioritization: a kernel consuming 40\% of GPU time ($p=0.4$) yields at most $S(\infty) = 1/(1-0.4) = 1.67\times$ whole-program speedup, regardless of how fast the kernel becomes. Conversely, a kernel at 5\% ($p=0.05$) caps at $1.05\times$. ScuffedKernels (\Cref{sec:scuffedkernels}) uses this to rank optimization targets by their \emph{Amdahl priority}: the product of potential speedup and absolute time saved.

\subsection{Gustafson's Law: Workload Scaling}

Gustafson's Law~\cite{gustafson1988} provides the complementary perspective for scaled workloads:
\begin{equation}
\label{eq:gustafson}
    S(n) = n - \alpha(n-1)
\end{equation}
where $\alpha$ is the serial fraction. For RDMA pipelines, increasing batch size scales the parallel portion while the serial overhead (connection setup, QP initialization) remains fixed. As a general example, with $\alpha = 0.05$ and $n = 4$ processors, $S(4) = 4 - 0.05 \times 3 = 3.85\times$---near-linear scaling. For our 3-GPU cluster, the applicable scaling depends on the measured serial fraction of the RDMA pipeline, which has not yet been determined.

\subsection{Arc Compute Pipeline Saturation}

Task matching for seamless GPU processing requires that compute kernels and RDMA transfers are pipelined such that neither the GPU nor the network is idle. The pipeline throughput is bounded by:
\begin{equation}
    T_{\text{pipeline}} = \max(T_{\text{compute}}, T_{\text{transfer}})
\end{equation}
When $T_{\text{compute}} \approx T_{\text{transfer}}$, the pipeline is saturated and both resources are fully utilized. ScuffedKernels achieves this by matching kernel optimization effort to the transfer-limited ceiling: optimizing a kernel below $T_{\text{transfer}}$ yields no further benefit.

%=============================================================================
\section{CUDA vs Triton: Kernel Abstraction Hierarchy}
\label{sec:cuda-triton}
%=============================================================================

GPU kernel development spans multiple abstraction levels. We use Triton for ScuffedKernels development, with CUDA paths planned for performance-critical kernels.

\subsection{CUDA Stack Layers}

CUDA provides four levels: Driver API (\texttt{cuLaunchKernel}, direct GPU control), Runtime API (\texttt{cudaLaunchKernel}, simplified launch), CCCL (CUB/Thrust template libraries), and C++ kernels (thread-level programming with explicit indexing and shared memory). CUDA Python wraps the Driver/Runtime APIs but cannot express kernels---Numba CUDA or Triton is required for kernel authoring.

\subsection{Triton: Block-Level Kernel Language}

Triton~\cite{triton2019} raises the abstraction from individual threads to \emph{blocks} of data. Key differences from CUDA:
\begin{itemize}
    \item \textbf{Block-level programming}: Operations on tiles (e.g., \texttt{tl.load}, \texttt{tl.dot}) rather than per-thread indexing.
    \item \textbf{Automatic memory management}: Triton handles shared memory allocation and bank conflicts.
    \item \textbf{Python-like syntax}: Kernels are decorated Python functions (\texttt{@triton.jit}).
    \item \textbf{Auto-tuning}: \texttt{triton.autotune} searches block sizes, warps, and pipeline stages.
\end{itemize}

The Triton compiler uses a multi-stage pipeline~\cite{triton-compiler-tips}: Triton IR (TTIR) $\to$ TritonGPU IR (TTGIR, with GPU-specific optimizations) $\to$ LLVM IR (LLIR) $\to$ backend-specific output (PTX/CUBIN for NVIDIA, AMDGCN/HSACO for AMD). Compiled kernels are cached under \texttt{\~{}/.triton/cache} keyed by source, parameters, and environment, enabling rapid iteration. Cross-architecture compilation via the \texttt{GPUTarget} API (e.g., \texttt{GPUTarget("cuda", 80, 32)} for SM\_80) permits artifact inspection without target hardware. For AMD targets, the compiled \texttt{.amdgcn} files expose register counts, spill information, and occupancy metrics directly.

Our ScuffedKernels baseline kernels (matmul, softmax, attention) are implemented in Triton for development speed, with CUDA fallback paths planned for performance-critical paths.

\subsection{NVSHMEM: GPU-to-GPU RDMA}

NVSHMEM~\cite{nvshmem} provides OpenSHMEM-like put/get semantics for GPU-to-GPU communication. Unlike NCCL (collective-focused), NVSHMEM supports one-sided operations from GPU kernels via \texttt{nvshmem\_float\_put} and \texttt{nvshmem\_float\_get}. However, NVSHMEM initialization is host-side only, and our Windows tower cannot participate in the symmetric heap. NVSHMEM is relevant for V100$\leftrightarrow$V100 transfers within Tower~2.

\subsection{Flash Attention: Triton vs CUDA Implementations}

Flash Attention~\cite{dao2022flashattention} reduces attention memory from $O(N^2)$ to $O(N)$ by tiling Q, K, V into SRAM-resident blocks and using the online softmax trick---maintaining running $(m, \ell)$ statistics and rescaling partial results per block:
\begin{equation}
    m_{\text{new}} = \max(m_{\text{prev}}, \tilde{m}), \quad
    \ell_{\text{new}} = e^{m_{\text{prev}} - m_{\text{new}}} \cdot \ell_{\text{prev}} + e^{\tilde{m} - m_{\text{new}}} \cdot \tilde{\ell}
\end{equation}
Block sizes are chosen as $B_c = \lceil M / (4d) \rceil$ (SRAM capacity $M$, head dimension $d$).

A Triton implementation study~\cite{aminediro-flash-attn} shows the optimization progression: naive (166~ms) $\to$ register accumulation (157~ms) $\to$ K-transpose to fix 6.3-way shared memory bank conflicts (34~ms, 4.9$\times$ total). The remaining bottleneck is MIO pipeline stalls from \texttt{exp()}/\texttt{max()} (44--50\% of time). Flash Attention 4~\cite{flash-attention-4} addresses this on Blackwell with warp specialization (producer/consumer warps), reporting 1.4~PFLOPS on B200. This illustrates Tier~5 of our optimization playbook (\Cref{sec:scuffedkernels}).

\subsection{AMD IRIS: Cross-Platform GPU Communication}

AMD's IRIS framework~\cite{amd-iris} brings SHMEM-like remote memory access (RMA) to ROCm GPUs within Triton kernels. IRIS is relevant for future cluster heterogeneity---if AMD GPUs join the cluster, IRIS provides a Triton-native communication path analogous to NVSHMEM. Our transport abstraction is designed to accommodate such backends.

The vLLM Semantic Router~\cite{vllm-semantic-router} classifies requests by semantic signal and routes to the best-suited model---optimizing for quality, not just throughput. For our cluster, this enables two-layer routing:

\end{enumerate}
The router's latency predictor can factor in RDMA transfer time from the tensor database, accounting for the different precision pipeline costs on each tower.

\section{Fine-Tuning and SAE Feature Steering}
\label{sec:finetuning-sae}
%=============================================================================

\subsection{Heterogeneous Training Pipeline}

The pipeline assigns roles by GPU capability: V100 pair runs forward/backward in FP16, the tensor database accumulates gradients into FP32 master weights, and the 5070~Ti receives BF16 weights for inference validation. This mirrors the Hetis approach~\cite{hetis2024}, which reported 2.25$\times$ throughput by assigning distinct pipeline stages to different GPU types.

\subsection{Sparse Auto-Encoder Feature Steering}
\label{sec:sae}

SAEs~\cite{cunningham2023sae} decompose activation $\mathbf{a} \in \mathbb{R}^d$ into sparse features $\mathbf{f} \in \mathbb{R}^D$ ($D \gg d$):
\begin{equation}
    \mathbf{f} = \text{ReLU}(W_{\text{enc}} \cdot \mathbf{a} + \mathbf{b}_{\text{enc}}), \quad
    \hat{\mathbf{a}} = W_{\text{dec}} \cdot \mathbf{f} + \mathbf{b}_{\text{dec}}
\end{equation}
Feature steering clamps feature $i$ to $\alpha \cdot \mathbf{f}_{\max,i}$, modifying behavior at inference time without gradient computation~\cite{eiffel-tower-llama}.

SAE feature vectors map naturally to the tensor database's \texttt{SparseVector}: 65,536 dimensions with $\sim$50 active features yields $\sim$400~bytes vs.\ 256~KB dense---51$\times$ compression, fitting within a single RDMA read.

\section{ScuffedKernels: RDMA-Aware Kernel Optimization}
\label{sec:scuffedkernels}
%=============================================================================

ScuffedKernels is our kernel optimization framework, providing systematic profiling, optimization, and validation of GPU kernels with RDMA-aware extensions. The implementation resides in \texttt{middleware/scuffedkernels/}.

\subsection{Architecture}

The system follows a four-stage pipeline:

\begin{enumerate}
    \item \textbf{Profiler} (\texttt{profiler.py}): Uses PyTorch CUDA profiler to extract per-kernel timing, then ranks kernels by Amdahl's Law priority (\Cref{eq:amdahl}). Kernels are scored by $(\text{speedup potential}) \times (\text{absolute time})$ to identify high-impact optimization targets.

    \item \textbf{Optimizer} (\texttt{optimizer.py}): Applies a 6-tier playbook, one tier at a time:
    \begin{enumerate}[label=T\arabic*:]
        \item Block/grid dimensions (SM occupancy)
        \item Memory access patterns (shared memory tiling, coalescing)
        \item Precision (TF32, FP16 Tensor Cores)
        \item Persistent kernels (fuse multi-stage into single launch)
        \item Architecture-specific (warp specialization, SM count tuning)
        \item RDMA-aware (compute-transfer overlap, prefetch depth)
    \end{enumerate}

    \item \textbf{Benchmarker} (\texttt{benchmarker.py}): 5-stage validation (smoke $\to$ shapes $\to$ stability $\to$ determinism $\to$ edge cases) followed by performance measurement. Includes RDMA-aware roofline analysis with a network bandwidth ceiling alongside the traditional compute and memory ceilings.

    \item \textbf{Orchestrator} (\texttt{orchestrator.py}): State machine driving the loop: profile $\to$ rank $\to$ optimize $\to$ validate $\to$ accept/revert. Each tier is validated before proceeding; failed tiers are reverted.
\end{enumerate}

\subsection{RDMA Extensions}

ScuffedKernels extends the classical roofline model with a \emph{network ceiling}. For RDMA-bound kernels (those waiting on network transfers), the achievable performance is limited by:
\begin{equation}
    P_{\text{net}} = \text{AI} \times B_{\text{network}}
\end{equation}
where AI is arithmetic intensity (FLOPs/byte) and $B_{\text{network}}$ is network bandwidth (12.5~GB/s for 100GbE). Kernels falling below this ceiling are network-bound, and compute optimization alone cannot help---prefetch depth and transfer overlap (Tier~6) are the appropriate interventions.

\subsection{Baseline Kernels}

Three baseline Triton kernels (\texttt{kernels/}) provide benchmarking targets: tiled matrix multiplication, online-normalized softmax, and Flash Attention-style tiled attention with the online softmax trick. Each has a NumPy reference implementation for correctness validation.

\section{ScuffedSearch: Autonomous RDMA Research Automation}
\label{sec:scuffedsearch}
%=============================================================================

ScuffedSearch provides a structured framework for systematic RDMA optimization experiments. The implementation resides in \texttt{middleware/scuffedsearch/}.

\subsection{Design Philosophy}

The core principle is \emph{one GPU, one file, one metric}:

\begin{itemize}
    \item \textbf{\texttt{prepare.py}} (IMMUTABLE): Defines the network topology (frozen dataclass), benchmark harness, and scoring function. Agents cannot modify this file---it ensures measurement consistency across experiments.

    \item \textbf{\texttt{optimize.py}} (MUTABLE): The only file agents modify. Contains tunable hyperparameters (batch size, QP depth, prefetch depth, wire format, transport mode, buffer pool size, congestion window) and the \texttt{optimize\_rdma\_transfer()} function.

    \item \textbf{\texttt{analysis.py}}: Tracks experiment outcomes (KEEP/DISCARD/CRASH), computes parameter contribution analysis, and maintains a JSON-serializable experiment history.
\end{itemize}

\subsection{Experiment Protocol}

Each experiment follows: (1)~make one change to \texttt{optimize.py}, (2)~run the harness across transfer sizes 1\,KB--16\,MB, (3)~score via $\text{score} = \sqrt{\text{throughput\_efficiency} \times \text{latency\_factor}}$ (geometric mean rewarding both bandwidth and latency), (4)~KEEP if $>$1\% improvement, else DISCARD, (5)~git commit for reproducibility.
